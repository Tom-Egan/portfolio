24MY Pivi Infotainment Update

Case Study 1
1

Problem

For the 24MY (2024 Model Year) mid-cycle refresh, Jaguar Land Rover removed a significant number of physical controls as part of a redesigned interior strategy. This resulted in several mission-critical vehicle functions needing to be safely and intuitively relocated into the touchscreen.

As one of the UX designers, I worked with a small design team to re-architect the existing Pivi infotainment system, adhering to strict legal, functional, and development constraints. Through multiple design iterations, interactive prototypes, and user tests, we delivered a production-ready solution that launched on the Range Rover Velar and later scaled across all JLR vehicles for 24MY and beyond.

2

Objective

The objective was to provide quick access to the high-frequency, safety-critical controls (previously operated by physical buttons) into the Pivi touchscreen. Affected features included:

  • Climate/HVAC
  • Heated/cooled seats
  • Volume
  • Drive modes
  • Defrost
  • Various vehicle settings (e.g. Auto Stop/Start)
3

Constraints

The primary design challenge was working within a tightly constrained solution space. Design modifications were limited to system sidebars and overlays, driven by functional requirements, legal regulations, and existing software architecture. These constraints defined the boundaries of the work and eliminated some UX patterns that we could explore.

  • Functional: Mission-critical systems such as climate and drive modes needed to be globally accessible in the system (i.e. can be accessed at anytime, regardless of which screen the user is on). In certain system states (e.g. while Apple CarPlay or Android Auto is active), the sidebars would be the only global entry point for these features.
  • Legal: Functions such as Rear Defrost came with their own set of rules which needed to be considered and also dictated their implementation.
  • Development: Integrating brand new, system level interactions into an existing software architecture meant that time and cost to develop factored into how much the system could/should be modified.
4

Approach

The What

A straightforward goal: Provide quick access to these high priority use cases, formerly physical controls, through the vehicle's touchscreen.

The Who

The work was shared between two designers, Mutlu Isik and myself, with each of us developing concepts in parallel before converging on a unified interaction model and design direction. We tackled feature sets one at a time in order to solve nuanced problems individually—such as temperature control behavior and climate layout— then integrated cohesively at a system level.

The How

  1. Design: One to two concepts created in parallel (in Sketch/Figma)
  2. Develop: High-fidelity, interactive prototypes of the design direction(s) created (in Proto.io)
  3. User Test Plan: Coordinate with the UX Research team
    • Determine interactions/user flows to examine
    • Define the user test tasks and user test plan
    • Conduct remote, A/B tests (if applicable) of the prototype(s)
  4. User Test Debrief: Review with our the UX Research team, sharing insights, recommendations, and next steps
  5. Iterate: Take the research recommendations and insights and return to Step 1 with a refined design direction
5

Role

I was responsible for end-to-end concept development, which included: wireframing, interaction design, high-fidelity prototyping, and research collaboration. I helped develop each of interactive prototypes used in all user tests and worked directly with UX Research teams in both Portland and the UK to define user test plans and refine designs post-testing.

With the JLR headquarters located in England, and our satellite office in Portland, Oregon, this was a global effort that involved regular collaboration with management and stakeholders in the UK.

Solutions

The following solutions demonstrate how specific information architecture and system-level interaction problems were solved within the tight spatial, legal, and technical constraints.

6.1: Problem Space

The first major problem to address was sorting out what and where everything would go into the sidebar, which boiled down to: the number of elements that needed to be added, exceeded the available space. Once placement of Rear Defrost and Max Defrost (legal requirements) was accounted for, many back-and-forth dicussions with manangement and stakeholders of which features to prioritize unfolded.

6.2: Contextual Buttons

  • Problem: Available space in the sidebar was insufficient for all required controls.
  • Solution: Introduced context-dependent buttons based on system state.
  • Impact: Enabled access to four high-priority features using only two sidebar positions, without increasing interaction depth.

To address space constraints, we introduced context-dependent buttons that adapted based on system state. One example was consolidating the previously separate Home and Apps buttons into a single contextual control.

  • If the user is on the home screen, display a shortcut button to open the App Drawer.
  • If the user is not on the home screen, display a shortcut button to return Home.

Note: The App Drawer could still be accessed via swiping from the left edge as a secondary method (Not quite as discoverable as the dedicated button, but still an available option for the user).

This approach was extended to the Phone shortcut. With Phone deprioritized, left without a permanent sidebar element, two edge cases still required attention: returning to active calls and returning to CarPlay or Android Auto. In these states, the Media button temporarily became a shortcut back to the active call or phone projection interface. Because call audio was already using the cabin speakers, Media access was unnecessary at that moment (i.e. show the user the right thing at the right time). This allowed four high-priority features to be served through two sidebar positions, freeing additional space without increasing interaction depth.

6.3: Combo Defrost Button

  • Problem: Available space in the sidebar was insufficient for all required controls, one-touch activation of Rear Defrost legally required.
  • Solution: Combine two related functions into a single switch, toggling between multiple modes.
  • Impact: Legal compliance maintained and enabled dual activation in a single press.

Another example of where creative problem solving became necessary was with the case of Rear Defrost and Heated Windshield. Functionally, these two features are related, in the sense that they both are turning on some sort of heated element in the glass. Given that, and the limited space in the sidebar, these two features seemed like logical candidates to pair together into one button. However, the legal requirement that was given to us—at any point, Rear Defrost must be able to be turned on within one press on the screen—presented a different challenge.

Several concepts were explored and ultimately rejected due to legal constraints requiring one-touch activation of Rear Defrost. The chosen direction was a toggle switch that cycles through four different states, while still complying with the Rear Defrost legal requirement. Our solution also created a beneficial byproduct: activating two controls on at once, in a single button press.

To reduce ambiguity, visual indicators were added to clearly distinguish front and rear control states and to signal that this control behaved differently from standard sidebar navigation. The interaction also adapted based on vehicle trim level—defaulting to a simple Rear Defrost toggle when Heated Windshield was not equipped.

6.4: Overlays

  • Problem: Broader feature sets needed quick access and to be globally accessible from any screen.
  • Solution: Use sidebar buttons as entry points to open an overlay, to allow for full-screen display.
  • Impact: Additional Climate, Volume, and driving-related controls accessible from any screen, reduced visual clutter during active overlays.

Using sidebar buttons as overlay entry points introduced the risk of mixed interaction models within the same visual grouping. This was mitigated by clearly delineating interaction zones and reinforcing behavior through distinct visual treatments and affordances.

Once an overlay is launched, the sidebars change to a more focused, empty state, removing unnecessary visual clutter. This was done to help communicate the context switch of having the overlay open, as well as provide the user a clearer visual path to their exit route (close button). One caveat to this stategy comes back to the previously mentioned legal requirements. Since one-touch activation of Rear Defrost and Max Defrost are required at all times, those two buttons needed to remain in place.

6.5: The Quick Slider

  • Problem: The new system lacked the ability for quick, large scale adjustments and quickly reach min/max values.
  • Solution: Add a vertical drag interaction to select sidebar buttons.
  • Impact: Rapid, low-precision adjustments supported. Interaction model reused across climate, drive modes, and volume

Introducing a secondary drag interaction to the overlay buttons enabled rapid adjustments, allowing for looser, less precise inputs from the user. Visual feedback reinforced state changes without requiring explicit confirmation.

The sidebar background provides blue feedback for additional confirmation when the selected temperature is in the lower/colder ranges. The sidebar background provides red feedback for additional confirmation when the selected temperature is in the upper/warmer ranges. To promote discoverability, a visual affordance—the Rail—was added to suggest a constrained drag interaction. Color cues reinforced directionality for temperature adjustment, and the same interaction model was later extended to Drive Modes and Volume for consistency.
